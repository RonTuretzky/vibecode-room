Model assignment: implement Anthropic, review OpenAI, verify by test

ORCH-A-04   Panopticon V0 implementation orchestration  ·  docs/planning/06-orchestration.md §4

The hard rule

The reviewing model family ≠ the implementing model family. A model reviewing its own family's output shares that family's blind spots — the same training-correlated failure modes survive both passes. Independent perspective is a first-class requirement of the validation bar, so we make family-divergence a structural invariant, not a hope.

The verifier's authority is the test result, never its opinion (the build-time analog of "invariants in code, not the LLM", ENG-A-04). It exists to confirm each blocking gate's test was capable of failing (the red run exists and is real) and that the green run passes — not to re-judge taste.

Safety-critical tickets

safety-execution-boundary-hook and shell-command-classifier are implemented with Opus 4.8 and additionally get an adversarial red-team pass from the reviewer family (Codex challenge against the read-safe allowlist — the §8.1.1 / FINDINGS RBG table). Note the product-runtime constraint from the POC (FINDING-1): the shipped safety processes must run on ClaudeCodeAgent (the PreToolUse hook is a Claude Code CLI settings.json mechanism, not a Smithers API). That constrains the product's agent, not the build's review — the cross-family review rule still applies to the code that implements the hook.

Alternatives

Same family implements and reviews (e.g. Opus writes, Sonnet reviews). Rejected — violates the hard rule. Shared blind spots survive both passes.
Self-review (implementer reviews its own diff). Rejected. No independence; "the agent said it's done" is never evidence.
Implement Anthropic, review OpenAI, verify by test (chosen). Chosen. Structural family-divergence + a test-authority verifier whose word is the gate result, not a vibe.

Verification

  • Cross-family-guard test: for every ticket assignment assert reviewer.family !== implementer.family. RBG: assign a Sonnet reviewer to an Opus implementer → guard rejects the assignment (else fails).
  • Verifier-independence test: assert the verifier agent is a distinct run/context from the implementer (different runId, fresh prompt). RBG: reuse the implementer context as verifier → independence assertion fails.
  • Review-recorded test: a land requires review.json with a cross-family verdict and verify.json with a passing gate re-run. RBG: drop review.json → land refused.
Fallback (R-ENG-09): if Fable 5 or an Anthropic tier is unavailable, the implementer defaults to Opus/Sonnet and the reviewer stays Codex; if Codex is unavailable, review falls back to a different Anthropic family member only as a logged exception — never same-instance self-review.
