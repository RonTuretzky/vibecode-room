Cue adapter owned hot path

BUILD   cue-adapter-and-policies  ·  2026-06-15

Decision

Keep command recognition authoritative in Cue's TextCue, and let the Vibersyn adapter emit the earcon from that Cue decision before any LLM or Smithers work. The adapter exposes a separate emitTextCueEarcon method for the hot path and a parity-checked pre-matcher hook only for the D2 fallback case.

Alternatives

Wait for the full Cue harness result. Rejected. That result can include slow LLM/tool work, so the earcon could be delayed by a path it must not wait on.
Use an independent adapter recognizer. Rejected. That would create a second routing authority outside Cue.
Emit from Cue's TextCue decision and assert fallback parity. Chosen. It preserves Cue authority and proves the earcon can fire within 300 ms while the LLM is delayed.
