Cue Smithers seam adapter

BUILD   probe-cue-smithers-seam  ·  2026-06-14

Decision

Panopticon V0 should connect Cue and Smithers with an async dispatcher behind Cue's MappedActionTool. The mapper should enqueue Smithers Gateway runs.create work and return a Cue action immediately. Gateway streamRunEvents frames are the V0 run-event stream source and are bridged back into Cue as observations carrying persisted upid, runId, and steeringWindowId correlation.

Why

The landed Cue interface executes MappedActionTool mappers inside the Cue ingest path, so awaiting the complete spawn path there would stall the Cue loop. The landed Smithers Gateway interface already proves runs.create, streamRunEvents, and durable restart behavior. Combining those surfaces with a deterministic dispatcher keeps LLM output limited to selecting the tool call while routing, persistence, reconnect, and correlation stay in code.

Alternatives

Await Smithers spawn inside the Cue mapper. Rejected. The RBG blocking run shows this stalls the Cue loop before spawn returns.
Shell out to the Smithers CLI from Cue. Rejected. The already-landed durable-runs probe selected Gateway mode for V0 process control.
Use an async Gateway dispatcher plus event bridge. Chosen. It exercises real Cue and real Smithers bidirectionally while preserving Cue loop responsiveness.
