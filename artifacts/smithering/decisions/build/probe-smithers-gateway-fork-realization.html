Smithers Gateway steering and fork realization

BUILD   probe-smithers-durable-runs  ·  2026-06-14

Decision

Vibersyn V0 process launch and voice steering should use Smithers Gateway mode and the Gateway submitSignal RPC. The adapter must not steer runs by shelling out to smithers signal. For process fork in the Gateway-backed V0 control plane, realize lineage as a fresh seeded Gateway run whose input carries parentId. Native Smithers time-travel fork exists as a CLI capability, but the installed Gateway surface does not expose a native fork RPC.

Why

The probe registered a workflow with the real Gateway, started runs with runs.create, streamed events with streamRunEvents, paused on a durable Signal, and resumed by calling submitSignal. A direct forkRun Gateway RPC returned METHOD_NOT_FOUND; a child run launched through runs.create preserved parentId in the checkpoint output.

Alternatives

Use detach mode and CLI signal commands for steering. Rejected. Detach mode requires separate resume handling and is not the real-time Gateway signal path Vibersyn needs.
Assume Gateway has native fork RPC. Rejected. The installed Gateway returned METHOD_NOT_FOUND for forkRun.
Use Gateway launch, stream, and signal APIs with seeded parent lineage. Chosen. It matches the proven runtime surface and keeps V0 process control behind one Gateway API path.
