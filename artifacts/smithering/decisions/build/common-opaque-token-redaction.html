Problem

The unknown-token detector handled provider prefixes, bearer values, JWTs, plus-sign tokens, long mixed alphanumeric values, and long alphabetic-only values. Follow-up review found three neutral-key gaps: opaque values using slash-only token alphabets, padded alphabetic values, and provider-prefixed alphabetic values.

Decision

Treat these as unknown secret candidates when they have token shape and enough length: padded opaque values, slash-only opaque values that do not look like paths, and short provider-prefix plus separator plus long opaque tail. The path guard avoids flagging ordinary filesystem paths in test logs while still redacting standalone token values. 

input meta: { opaque: "provider-prefix + long opaque tail" }
output meta: { opaque: "redacted marker" }

Alternatives

AlternativeVerdictReason
Require a digit in every unknown tokenRejectedOpaque provider tokens can be alphabetic.
Redact every long separator-bearing wordRejectedToo many build paths and ticket ids contain separators.
Redact padded, slash-only non-path, and provider-prefixed opaque valuesChosenCloses the demonstrated gaps while keeping source and evidence scans usable.

Verification

src/providers/credentials.test.ts and test/e2e/secret-scan.e2e.ts plant these opaque token shapes in trace metadata and probe reports. The captured red runs fail before the detector fix; the green run passes with zero bundle scan findings.
