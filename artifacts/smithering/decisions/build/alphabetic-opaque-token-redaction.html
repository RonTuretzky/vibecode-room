Long alphabetic opaque values redact fail closed

subscription-credentials-redactionSEC-1

Problem

The unknown-token fallback redacted provider prefixes, JWTs, bearer values, separator-heavy values, and long alphanumeric values with digits. A long alphabetic-only opaque value under a neutral metadata key could still be emitted unchanged.

Decision

Treat alphabetic-only values of at least 32 characters as unknown secret candidates. This keeps the detector fail closed for opaque subscription tokens while avoiding common identifier-length noise in source and reports.

input meta:  { opaque: "long alphabetic token material" }
output meta: { opaque: "redacted marker" }

Alternatives

AlternativeVerdictReason
Require a digit in every unknown tokenRejectedThe adversarial challenge showed a fail-open alphabetic-only class.
Redact every long word at 24 charactersRejectedToo noisy for source and report scans.
Redact alphabetic-only values at 32 charactersChosenCloses the opaque-token gap with limited false positives.

Verification

src/providers/credentials.test.ts and test/e2e/secret-scan.e2e.ts plant alphabetic-only opaque values in trace and probe metadata. The captured red run fails before the detector fix; the green run passes with zero bundle scan findings.
