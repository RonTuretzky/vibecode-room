Credential-shaped metadata keys are redacted before emission

subscription-credentials-redactionSEC-1

Problem

The redaction filter already transformed secret-shaped string values, but adversarial metadata can put a credential-shaped token in an object property name with a harmless value. JSON emission still persists property names, so source values and keys must be treated as equally scannable output.

Decision

Redact object keys with the same secret-shape detector used for values before any trace, probe, or report is emitted. Nested value redaction still uses the original key path so secret-named fields keep fail-closed value redaction. If multiple keys redact to the same marker, append a non-secret numeric suffix to avoid dropping data.

input meta:  { credential-shaped-key: "harmless" }
output meta: { "redacted marker": "harmless" }

Alternatives

AlternativeVerdictReason
Only redact valuesRejectedProperty names are persisted in JSON and scanned by SEC-1.
Drop matching key/value pairsRejectedLoses useful non-secret diagnostics and makes traces harder to reconstruct.
Redact keys and preserve valuesChosenKeeps diagnostics while making emitted JSON fail closed.

Verification

src/providers/credentials.test.ts now includes a red-before-green metadata-key case, and test/e2e/secret-scan.e2e.ts plants credential-shaped keys into both session trace metadata and probe-report metadata. The ticket bundle secret scan reports zero findings.
