P-CUE real Cue API differences

BUILD   probe-cue-substrate  ·  2026-06-14

Decision

Treat P-CUE as green for REQ-1/3/5/6/7 substrate validation because the real Cue source builds and the required voice-loop primitives are executable. Record three adapter-owned extensions: cadence cooldown around IntervalCue, an EventSource SSE bridge if the board insists on SSE, and Panopticon transcript/correlation normalization around Cue observations.

Facts From The Probe

The source build resolved github.com/jameslbarnes/cue at commit 2dbccc023863f2f0563e06c4d0a6aa44cff3988b. The probe exercised real exports from @cue/core and @cue/server, not README-derived shapes.

Confirmed:
TextCue, SpeakerWordCue, IdleCue, WordCountCue, IntervalCue.intervalSeconds,
TextCue.cooldownSeconds, observe.pass, CueHarness, SemanticProgram,
MappedActionTool, qwen-asr JSON transcription ingress, voxterm, JSONL recording,
HTTP state/agent/observations, WebSocket events, LLM/output/VLM provider slots.

Differences:
IntervalCue does not directly expose cooldownSeconds.
The live event route is WebSocket, not native text/event-stream SSE.
Current Cue source also exports deepgramTranscriptionProvider; qwen-asr remains
the JSON receiver shape ENG-T-10 should feed when Panopticon owns ASR.

Alternatives

Fail P-CUE for any API naming difference. Rejected. The differences are thin adapter responsibilities already allowed by D2; blocking would not improve the design.
Hide the differences and report only confirmed primitives. Rejected. Dependent tickets need the exact extension list before building on the seam.
Pass the substrate probe and record owned extensions. Chosen. The real primitives work, and the required local work is explicit and testable.
