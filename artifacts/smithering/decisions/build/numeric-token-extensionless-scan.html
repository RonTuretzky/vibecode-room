Numeric provider tokens and extensionless files fail closed

subscription-credentials-redactionSEC-1

Problem

Follow-up challenge found two remaining fail-open paths: provider-prefixed opaque values made from long numeric groups separated by hyphens, and extensionless files under trace/log/report roots. Both can carry durable secret material even when their metadata key or filename is otherwise neutral.

Decision

Treat a short alphabetic provider prefix followed by two or more long numeric groups as an unknown secret candidate. Also scan every regular file under a requested root instead of limiting scans by filename suffix. 

input meta:  { opaque: "provider-prefix + numeric groups" }
output meta: { opaque: "redacted marker" }

Alternatives

AlternativeVerdictReason
Only redact mixed alphanumeric groupsRejectedOpaque provider tokens can use numeric account or workspace segments.
Keep the scan filename suffix listRejectedTrace and report trees can contain extensionless logs or generated report files.
Redact provider-prefixed numeric groups and scan all filesChosenCloses the demonstrated SEC-1 gaps without adding a credential-provider abstraction.
