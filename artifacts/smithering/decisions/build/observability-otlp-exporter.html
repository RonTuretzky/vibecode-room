OTLP export for Smithers agent calls

BUILD   observability-trace-and-board  ·  2026-06-15

Decision

Implement src/obs/otel.ts as a small OTLP/JSON exporter with GenAI semantic-convention attributes, injected fetch, and no global OpenTelemetry SDK registration. It posts to the configured Langfuse OTLP endpoint when present and otherwise no-ops.

Why

P-OTEL is informational and observability is off every voice critical path. The local package already pulls OpenTelemetry transitively through Smithers, but adding global SDK setup here would introduce process state and shutdown behavior that the board and causal reconstruction do not need. A narrow OTLP exporter still proves the payload contract against a Langfuse-compatible endpoint while keeping the voice loop independent of backend availability.

Alternatives

Install and configure a global OpenTelemetry SDK provider. Rejected for V0. It increases lifecycle surface for an informational path.
Skip OTLP and rely only on JSONL. Rejected. The ticket explicitly asks for Langfuse export via OTLP.
Use a narrow OTLP/JSON exporter with GenAI attributes. Chosen. It is easy to test with a local HTTP receiver and remains off-path.
