Model assignment: implement Codex, verify Sonnet, review Codex + Opus, plan Opus

ORCH-A-04   Vibersyn V0 implementation orchestration  ·  docs/planning/06-orchestration.md §4
New matrix. The prior decision used a complexity-based assignment where Anthropic implements (Sonnet/Opus by size) and OpenAI Codex reviews. That is replaced by fixed per-role assignments below. The cross-family principle still holds — the default implementer is now OpenAI/Codex, and the cross-family Anthropic check is the Opus reviewer.

Cross-family still holds

The implementer is OpenAI/Codex; the cross-family Anthropic check is the Opus reviewer. Review runs both Codex 5.5 and Opus 4.8, so every diff gets an in-family adversarial pass (Codex reviewing Codex's implementation) and an independent cross-family perspective (Opus). The cross-family invariant — a reviewer of a different family than the implementer — is satisfied by the Opus reviewer. A model reviewing only its own family's output shares that family's blind spots; pairing the Codex implementer with an Opus reviewer keeps independent perspective structural, not hoped-for.

The verifier's authority is the test result

Verification is Sonnet 4.6, and the verifier's authority is the test result, never its opinion (the build-time analog of "routing authority in code, not the LLM", ENG-A-04). The verifier confirms each blocking gate's test was capable of failing (the red run exists and is real) and that the green run passes — it does not re-judge taste.

Alternatives

Complexity-based: Anthropic implements (Sonnet/Opus by size), Codex reviews (prior plan). Replaced. Superseded by fixed per-role assignments; the default implementer is now Codex, not Anthropic.
Single-family review (only Codex, or only Opus). Rejected. Review runs both so a diff gets in-family adversarial review and a cross-family independent check.
Implement Codex 5.5, verify Sonnet 4.6, review Codex 5.5 + Opus 4.8, plan Opus 4.8 (chosen). Chosen. Fixed roles, cross-family check via the Opus reviewer, test-authority verifier. (Fable 5 is a TODO.)

Verification

  • Role-assignment test: for every ticket, assert implement = gpt-5.5, verify = claude-sonnet-4-6, review includes both gpt-5.5 and claude-opus-4-8, plan = claude-opus-4-8. RBG: assign Opus as the implementer → assignment guard rejects it (implement must be Codex).
  • Cross-family-guard test: assert the review set contains a reviewer of a different family than the implementer (the Opus reviewer vs. the Codex implementer). RBG: drop Opus from review, leaving only Codex → cross-family assertion fails.
  • Verifier-independence test: assert the Sonnet verifier is a distinct run/context from the implementer (different runId, fresh prompt). RBG: reuse the implementer context as verifier → independence assertion fails.
  • Review-recorded test: a land requires review.json with both reviewers' verdicts and verify.json with a passing gate re-run. RBG: drop review.json → land refused.
Fallback: if Codex is unavailable, implementation cannot proceed for that ticket (Codex is the implementer); if an Anthropic tier is unavailable, the Opus reviewer/planner falls back to another Anthropic family member as a logged exception — never same-instance self-review.
