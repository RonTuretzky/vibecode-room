Tiered vocabulary (always-hot vs. state-gated): deferred for V0

D-DD-24   Panopticon V0 — final pass  ·  docs/planning/02-design.md §5.1
Superseded: the prior pass split the vocabulary into an always-hot tier (rare words matched anytime) vs. a state-gated tier (natural words matched only inside a narrow window), with a phoneme-distance collision bar scaled to false-trigger cost. That tiering is NOT implemented for V0. Cue (Etheria's voice library) handles wake/keyword activation, so we do not build a custom two-tier matcher. Revisit later only if it becomes a real problem.

The original puzzle

The command set lists "Yes" / "Accept" as acceptance words — conversational words that, if matched at any moment, could misfire. The earlier solution was to distinguish when a word is live (tiering). With Cue owning keyword activation, we no longer design or test that ourselves for V0.

Decision

Do not implement a tiered always-hot/state-gated vocabulary for V0. Keyword/wake activation — including not firing on incidental conversational uses — is delegated to Cue. Magic words for commands still exist (yes, magic words — see ENG-A-16), but the bespoke two-tier matcher, phoneme-distance collision bar, and "cost-scaled rarity" machinery are removed as a V0 requirement / acceptance criterion.

Why defer

The tiering existed to solve a problem we assumed we had to solve in-house: keeping rare words live everywhere while allowing natural words inside windows, all without misfiring. That is exactly the robustness Cue provides. Building our own tier system duplicates the library and adds collision-guard tests that prove a library capability we already trust. If real-world use later shows a specific word misfiring, we revisit — but not as V0 scope.

Alternatives

A. Build the two-tier matcher now (prior plan). Rejected for V0: duplicates Cue's activation handling; over-engineered for a problem the library owns.
B. Make everything always-hot in our own matcher. Rejected: still a bespoke matcher we don't need.
C. Defer tiering; delegate activation to Cue (chosen). Chosen. Ship V0 fast; revisit only if a misfire problem actually appears.

Verification

  • Integration: assert command magic words ("accept", callsigns, "mute"/"unmute", "halt") are wired into Cue and toggle the right actions. RBG: mis-wire a word → wrong action fires → fails. We verify wiring into Cue, not a custom tier matcher.
  • E2e: un-addressed talk does not trigger commands; addressed magic words do — robustness provided by Cue, observed end-to-end rather than asserted against an in-house collision guard.
  • Observability: command.recognize{phrase, matchedCommand|null}.
