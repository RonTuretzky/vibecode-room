Unmuting while muted: the "unmute" word (via Cue) + an on-screen button

D-DD-22   Panopticon V0 — final pass  ·  docs/planning/02-design.md §12
Superseded: the prior pass proposed building a bespoke on-device keyword spotter that stayed running while muted to catch the unmute word. That custom component is cut. The voice library (Cue, by Etheria) already handles always-on keyword listening even while cloud transcription/suggestions are paused, so we don't build our own spotter.

The question

REQ-2 AC2.3: while muted, no observations are produced and audio is not fed into the suggestion/routing pipeline.  vs.  D1: voice is the sole operational modality.  →  If nothing is feeding the pipeline while muted, how does the room unmute by saying "unmute"? Answer: Cue keeps listening for its keywords even while the pipeline is paused — that is what the library is for — and the screen always shows an unmute button as a second path.

Decision

There are two ways to unmute:
  1. Say "unmute" — Cue's always-on keyword listening catches it even while the suggestion/routing pipeline is paused, and emits mute.released.
  2. Press the on-screen "unmute" button — always available, emits the same mute.released event.
"Muted" means: stop feeding audio into the suggestion/routing pipeline (no observations, nothing streams to cloud transcription/suggestions). Cue still listens for "unmute"; the button is always there too. We do not build a custom on-device spotter — Cue owns the keyword listening.

Why this preserves AC2.3 / NG-6

While muted, the suggestion/routing pipeline is closed: no observations (the thing AC2.3 forbids) and no raw audio or transcript is written (NG-6 / AC1.3). Cue's keyword listening is a library capability, not "the ASR" — its only effect in this state is to flip the mute state back when it hears "unmute." So "no observations while muted" holds literally — the cloud suggestion path that would produce observations is closed for the entire interval.

Alternatives

A. Build a bespoke on-device spotter. Rejected / cut: Cue already provides always-on keyword listening — building our own duplicates the library and adds a probe, a native-module dependency, and degradation logic we don't need.
B. Keep the full pipeline running but "ignore" everything except the unmute word. Rejected: audio still flows into the suggestion path and observations still exist — violates AC2.3 and NG-6 outright.
C. "unmute" word (Cue) + on-screen button (chosen). Hands-free and screen-clickable, nothing leaves the room while muted, AC2.3/NG-6 intact, no custom component to maintain. Chosen.
