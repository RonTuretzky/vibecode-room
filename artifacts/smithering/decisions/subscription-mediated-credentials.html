Model calls use the host's logged-in Codex + Claude subscriptions — no raw keys, no credential-provider machinery

ENG-A-15PRD §6Simplified

Superseded / simplified. The prior decision routed every model call through a Smithers SubscriptionCredentialProvider abstraction, defaulted the hot-loop model to Claude Haiku-4.5 via subscription, kept a fallback provider as a contingency, and added an A-LLM-SUB probe. That elaborate credential-provider abstraction, the named hot-loop model/fallback machinery, and the provider probe are dropped.

The decision

Assume the host machine running this is already logged in to its OpenAI Codex and Anthropic Claude subscriptions — the local CLIs / subscriptions are available — and model calls simply use those. There are no raw API keys in source and no elaborate credential-provider abstraction. Model choice follows the model-assignment matrix (ORCH-A-04): implement = Codex gpt-5.5, verify = Sonnet claude-sonnet-4-6, review = Codex gpt-5.5 + Opus claude-opus-4-8, plan = Opus claude-opus-4-8 — reached through the host's logged-in subscriptions, not a key.

Why simplified

The whole credential-provider layer existed to mediate raw keys safely. If we instead assume the host already has logged-in Codex and Claude CLIs/subscriptions, the keys never live in our source or env at all — the host owns them — so the abstraction has nothing to abstract. We keep the one principle that matters (no raw keys in source) and delete the machinery.

One simple path

host machine (already logged in)
   ├── OpenAI Codex CLI / subscription   → implementation + review (gpt-5.5)
   └── Anthropic Claude subscription     → verify (sonnet-4-6) · review/plan (opus-4-8)

model calls use the host's logged-in subscriptions   (no raw key in our source or env)
   no SubscriptionCredentialProvider, no fallback-provider hot loop, no A-LLM-SUB probe
