Worktrees & concurrency: one worktree per ticket, DAG antichains drive parallelism

Decision — worktree layout

One jj/git worktree per ticket, co-located in the target repo (.), branched off the current main tip once all of the ticket's dependsOn ancestors have landed.
KnobChoice
Granularityper ticket (the stable kebab id is the durable Smithers task id), not per phase, not per file
Branch / pathbuild/<ticketId> in worktree .smithers/wt/<ticketId>
Basemain at the ticket's dependency-closure tip; Smithers worktree runs auto-rebase on resume (conflicts materialized by jj, not silently dropped)
Lifetime= ticket lifetime; removed after land (or kept if it still carries an unresolved jj conflict for the worker to fix)
Probe worktreesdisposable; write only under poc/ and artifacts/smithering/probes/ — disjoint paths, so they parallelize freely and land via a trivial fast lane (no src conflicts)
Isolation is the point: parallel tickets never touch the same working copy, so the only place edits meet is the serialized merge lane (ORCH-A-02), where jj's first-class conflict handling makes a bad rebase a recoverable bounce rather than a corrupt land.

Decision — concurrency cap

Max 6 concurrent ticket workers; merge lane depth 1; probe phase may burst to 8 (probes are disjoint-path and external-API-latency-bound, not CPU-bound, and do not merge src).

Why 6, grounded in the actual DAG rather than a round number:

  • The widest ready-set (antichain) in the graph is ~6: Phase-0's fan-out (record-replay-harness, provider-interface-doubles, trace-processor-observability, probe-suite-harness) is ~4, and Phase-1's independent probes (probe-cue-substrate, probe-asr-deepgram, probe-hot-loop-llm-subscription, probe-streaming-tts, probe-smithers-durable-runs, probe-keyword-spotter) is ~6. Beyond that the DAG narrows. A cap above 6 buys idle workers, not speed.
  • Each worker is a heavy durable run (implementer + cross-family reviewer + verifier, each running bun test / tsc, some spawning real Smithers child runs). 6 keeps us under the host's Smithers run-slots — the build-layer analog of §10.1's pre-spawn resource check — leaving headroom for the postsubmit live-e2e / latency / durability suite so it isn't starved.
  • Because the merge lane is depth-1, more than ~6 in-flight just multiplies rebase churn: every land invalidates the others' base. 6 matches throughput to the lane's drain rate.

Scheduling

ready(t)  := every dep in t.dependsOn has LANDED on main
            AND every blocking probe t depends on is GREEN (pre-build tier)
schedule  := take up to (6 - inFlight) tickets from ready(), topo-rank order
land      := depth-1 serialized lane; rebase onto tip; pre-merge gate; ff or bounce

Alternatives

One worktree per phase (shared by all tickets in it). Rejected. Re-introduces in-worktree contention the per-ticket split exists to remove.
Unbounded workers (one per ready ticket). Rejected. Exceeds host run-slots and starves postsubmit; churns the depth-1 lane.
Per-ticket worktree, cap 6, antichain scheduling (chosen). Chosen. Matches the DAG's real width, isolates edits, and keeps the lane drainable.

Verification

  • DAG-acyclic test: assert tickets.json dependsOn is acyclic and every ref resolves. RBG: add a back-edge → cycle detector fails.
  • Antichain-schedule test: never schedule a ticket whose deps haven't landed; never exceed 6 in-flight. RBG: raise the cap to 99 with a 7-wide ready-set → over-cap assertion fires.
  • Worktree-isolation test: two parallel workers writing the same path land without clobber — the second rebases and either merges cleanly or bounces with a jj conflict. RBG: share one working copy → concurrent-edit corruption detector fails.
jj specifics: co-located jj+git so Smithers durability snapshots map onto jj operations; jj revert is the postsubmit back-out primitive (ORCH-A-02); auto-rebase-on-resume means a resumed worker re-reads disk and re-derives state — it never trusts in-memory context (ORCH-A-07).
