Tests: run ALL of them in both pre-submit and post-submit — don't design tiers

ORCH-A-03   Vibersyn V0 implementation orchestration  ·  docs/planning/06-orchestration.md §3
Simplified. The prior decision designed a three-tier split — a deterministic hermetic subset gates the merge (pre-merge), with everything slow/real-world deferred to postsubmit, plus a pre-build probe tier. That tiering is dropped. Run ALL tests in BOTH pre-submit and post-submit. Don't design tiers.

The decision

Run the entire test suite both pre-submit (before a land) and post-submit (on main after a land). No pre-merge "deterministic subset," no postsubmit-only bucket — the same full suite runs in both places. Assume tests are fast. We don't plan around slowness: don't build tier-partitioning machinery. (The probe gate — validate-before-build — still exists as its own thing; see ENG-A-06. This decision is about the unit/integration/e2e test suite, not the dependency probes.)

Why simplified

Tiering existed to keep lands fast by gating on only the cheap deterministic checks and pushing slow real-world ones to postsubmit. That is exactly the kind of premature optimization the ship-fast posture avoids: it adds a partition to design, test, and keep consistent. Assume the suite is fast enough to run in full on every land and again on main. If a specific test later proves genuinely slow, whoever is monitoring can move it out — but we don't design for that up front.

What runs where

WhenWhat runs
Pre-submit (before a land, on the rebased tip)ALL tests — unit, integration, schema, tsc --noEmit, smoke, lint, e2e — the full suite.
Post-submit (on main after each land)ALL tests — the same full suite again, against the integrated tip.

Every test carries its recorded red-before-green. Nothing is permanently skipped because nothing is partitioned out in the first place.

Alternatives

Three-tier split: deterministic subset pre-merge, full suite postsubmit, probes pre-build (prior plan). Dropped. Premature optimization: a partition to design and keep consistent, built on the assumption that tests are too slow to run in full.
Pre-merge subset with no postsubmit backstop. Rejected. Would silently skip checks forever — but moot here, since we run everything in both places.
Run the full suite in both pre-submit and post-submit; assume tests are fast (chosen). Chosen. No tier machinery; maximum confidence on every land and every integration.

Verification

  • Full-suite-both-places test: assert the pre-submit run and the post-submit run invoke the same full set of tests. RBG: drop a test from either run → parity assertion fails.
  • No-tiering test: assert there is no pre-merge "subset" partition that excludes any test. RBG: introduce a subset that skips a test pre-submit → no-tiering assertion fails.
  • Probe-precedence test: a ticket whose dependency's blocking probe is unrun/failed cannot start (the probe gate is independent of the test suite). RBG: mark P-CUE failed → assert cue-adapter-and-policies is not scheduled.
Note: if a test ever turns out to be genuinely slow, whoever monitors the run can move it out of the per-land path — but that is a reactive fix by the operator, not a tier we design and build in advance.
