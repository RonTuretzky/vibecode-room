CUT: one mode for V0 — run "dangerously" (run to completion)

ENG-A-11REQ-11Scope cut

Superseded / CUT. The prior decision defined three execution modes — Safe (default, gates mutating tool classes), Explicit (gates every action), and Dangerous (bypasses the gate) — with Explicit/Dangerous opt-in by warned voice command and a reset-to-Safe at session end. That whole mode-switching system is removed.

The decision

For V0 there is one mode: run "dangerously" — run-to-completion, no per-step approval. We do not build Safe/Explicit/Dangerous mode switching, the setMode action, the ExecutionMode enum, the warned voice opt-in, or the session-reset behavior. The system simply runs the work to completion.

Why cut

Three permission postures plus voice opt-in plus warning read-backs plus session-reset is exactly the gating machinery the V0 posture (cut scope, ship fast, run dangerously) removes. There is no Safe-vs-Explicit distinction to maintain because there is no per-action gate at all (ENG-A-08 is cut). If we want safety later, we sandbox the whole process — isolate it — rather than gate individual actions through permission classes.

Verification

  • Single-mode test: a fresh process runs to completion with no per-step approval and no mode selector (RBG: re-introduce a Safe-mode gate → the run-to-completion e2e fails).
  • No mode-switch test: assert no setMode action / ExecutionMode switching exists.
  • E2e: a destructive act proceeds without a gate; there is no session-reset to a "safe" mode because there is only one mode.
Note: sandboxing the whole process is the deferred safety lever (revisited later), not permission modes.
