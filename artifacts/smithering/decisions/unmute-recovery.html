Unmute = say "unmute" (via Cue) or press the on-screen unmute button

ENG-A-13REQ-2 / D1Resolves R10

Superseded: the prior decision built a bespoke on-device "spotter" as the sole operational unmute, promoted P-SPOTTER to a blocking P0 probe, and made spotter-down recovery an emergency kill-all + clean restart. All of that is removed. Cue (Etheria's voice library) already listens for keywords while the pipeline is paused, so there is no custom spotter, no P-SPOTTER probe, and no teardown-restart recovery path.

The decision

There are two ordinary unmute paths, either of which works at any time while muted:

  • Say "unmute" — Cue's always-on keyword listening catches it even while the suggestion/routing pipeline is paused, and emits mute.released.
  • Press the on-screen "unmute" button — always rendered, emits the same mute.released.

Because there are two independent paths and one of them (the button) never depends on audio or any third-party model, the room can never get trapped muted. We do not repurpose the REQ-14 emergency control as an unmute, and there is no kill-all-and-restart recovery for "unmute is broken" — the button is always there.

Why — the R10 finding, re-resolved

R10 originally asked how a disabled/failing custom spotter degrades. With the spotter cut, the question dissolves: unmute is (a) a keyword Cue already listens for, and (b) a screen button that is always present. Neither requires a bespoke component or an emergency teardown.

The flow

MUTED:   "mute" → pause the suggestion/routing pipeline + earcon + one-word "Muted"
              while muted: Cue still listens for "unmute" (library capability — no custom spotter)
                          the on-screen "unmute" button is always shown
         say "unmute"  → Cue → mute.released → pipeline resumes        ← operational unmute (hands-free)
         click button  → mute.released → pipeline resumes              ← always-available unmute

REQ-14's non-voice emergency control stays exactly what the PRD says — a kill-all that halts every process and ends the session — and is never reused as an in-session unmute. It is no longer part of any unmute-recovery story, because there is nothing to recover from.

Verification

  • two-unmute-paths test — both "unmute" (wired into Cue) and the on-screen button emit mute.released and resume the pipeline (RBG: disable one path → assert the other still unmutes; disable both → room trapped muted → fails).
  • no-unmute-verb-on-emergency test — the REQ-14 emergency control exposes no resume/unmute path (RBG: wire it to resume streaming → AC14.2 scope test fails).
  • mute-announce test — "mute" emits earcon + one-word "Muted" (AC2.4).
Note: no P-SPOTTER probe exists anymore. We trust Cue for keyword listening and verify only that "unmute" is wired into it and that the button always renders and unmutes.
