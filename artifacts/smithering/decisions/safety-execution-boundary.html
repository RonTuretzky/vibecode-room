CUT: no destructive read-back / confirm gate — we run to completion

ENG-A-08REQ-11Scope cut

Superseded / CUT. The prior decision built an in-run PreToolUse hook + safe-executor that intercepted every destructive tool call, held it deny-by-default, routed a spoken read-back/confirm across the seam, and aborted via a 25 s dead-man timer if no "confirm" arrived. That entire mechanism — the hook, the safe-executor, the spoken read-back gate, and the timer — is removed.

The decision

For V0 we run to completion. We do not build a heavy per-action approval / spoken read-back gate, and there is no dead-man timer holding destructive tool calls. You shouldn't need to approve often — minimize approvals. Where a confirmation is genuinely needed, the voice library (Cue, by Etheria) already handles it; we don't reimplement a confirmation layer inside the Smithers run.

Why cut

The overall V0 posture is: cut scope, ship fast, run dangerously, trust the library. A per-tool-call interception layer with deny-by-default classification and a spoken-confirm round-trip is exactly the kind of heavy machinery that posture removes. The product is meant to run autonomously to completion; constant per-action gating fights that goal. If we want real safety later, we sandbox the whole process (ENG-A-11), not gate individual actions through permissions.

What stays

Routing authority in code (ENG-A-04) is unaffected: deterministic code still decides where an utterance routes. What is gone is the execution-time destructive-action gate — there is no read-back, no confirm-before-execute, no timer.
