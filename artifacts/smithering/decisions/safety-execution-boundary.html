Safety is enforced at the agent's tool-call boundary

ENG-A-08REQ-11Resolves R1 (critical)Probe: P-HOOK

The decision

The destructive-action read-back/confirm gate (REQ-11) lives in an in-run PreToolUse hook + safe-executor registered inside every Panopticon Smithers process — it intercepts the actual tool call before execution, classifies it deny-by-default, holds it, and routes the read-back to voice across the seam. The dead-man timer (25 s) lives in the hook, so a missed/mis-heard "stop" aborts rather than executes. It is probe-gated by P-HOOK before any code trusts it.

Why — the R1 finding

"If destructive work happens inside a Smithers run, Panopticon cannot guarantee it sees the action before execution. The static verb list also misses many dangerous operations."

The destructive act is not the steering instruction the human speaks — it is the tool call the agent emits later, inside the durable run, possibly several reasoning steps after (and not lexically related to) any instruction. The dispatcher only sees actions flowing out of Cue; it never sees Bash("rm -rf …"), a file overwrite, git push --force, or a DB DROP that the agent decides to run on its own.

Mechanism

agent emits tool call {tool, args}
   │
   ▼  PreToolUse hook (in the Smithers run)
classify(klass)  ── read ───────────────▶ execute (pass)
   │ fs-write|fs-delete|shell|vcs-push|db-mutate|net-mutate|unknown
   ▼  (Safe mode: gate;  Explicit mode: gate ALL;  Dangerous mode: bypass)
HOLD the call  ──▶ ApprovalRequest{gateId, readback} ──seam──▶ Cue ──▶ TTS
   │                                          "I'm about to delete X. Say 'confirm'."
   ├── room says "confirm" ──▶ dispatcher.approve(gateId) ──▶ release ──▶ execute ONCE
   └── 25 s dead-man timer  ──▶ abort (file unmodified) ──▶ E5 + safety.resolution{timedOut:true}

Deny-by-default is the key strengthening over a verb list: anything in a mutating or unknown class is gated. The verb/pattern fast-path (rm, drop, force-push, truncate, overwrite) is an optimization on top, never the authority.

Verification (the evidence, not the claim)

  • P-HOOK (P0, blocking): against the real Smithers API — register a PreToolUse hook, have an agent attempt a destructive tool call, assert the hook fires before execution, can hold the call, and that approve / deny / timeout resolve correctly with the file verifiably unmodified on deny/timeout.
  • Unit: hook-intercept (held, not executed; RBG: bypass → file modified before approval → fails); deny-by-default (unknown class gated; RBG: default unknown→allow → fails); dead-man-timer (mocked clock → abort); fuzz (wrong gateId, double-confirm → never double-execute).
  • E2e: instruct toward a destructive act → blocks (file unmodified while held); withhold confirm → aborts; confirm → executes exactly once; explicit mode gates a non-destructive action; dangerous mode (after warning) bypasses.
Surfaced to the gate: if P-HOOK fails, REQ-11 is not enforceable as designed — fall back to the constrained-tool-layer wrapper (R-ENG-02) or revisit the optimistic posture with the PRD gate. The safety guarantee is treated as a lie until P-HOOK is green.
