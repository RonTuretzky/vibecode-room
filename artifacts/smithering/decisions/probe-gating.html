Dependency→probe gating & blocking classification

ENG-A-06   Panopticon V0 engineering  ·  validate-before-build  ·  docs/planning/03-eng.md §14, §15

The rule

Every non-framework third-party dependency is exercised against the real API by a probe that asserts the exact behavior we rely on, before any product code is built on it. A probe that could fail and passed is the evidence — docs and memory are not. Every dependency maps to ≥1 probe; every probe is blocking unless explicitly justified. React / standard libraries are exempt.

How "blocking" is decided

A probe is blocking when a failed result forces an architectural redesign or makes a hard requirement unprovable. It is non-blocking only when a failed result has a pre-designed graceful fallback with no architectural impact — and that fallback is named.

Verification of the gate itself

  • The probe is the test — each assertion must be able to fail (feed a below-threshold input, assert the cue does not fire; feed a passing one, assert it does). Record red-before-green for every assertion.
  • Probe scripts live under poc/; results under artifacts/smithering/reports/. All are currently UNRUN — nothing is confirmed.
