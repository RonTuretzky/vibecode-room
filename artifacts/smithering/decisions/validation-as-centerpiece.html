Validation & observability as the document's centerpiece

D-DD-25   Vibersyn V0 design — final orchestrator pass  ·  doc: docs/planning/02-design.md

Decision

Restructure the design doc so the verification plan is the centerpiece, not an appendix. Every design section (§2–§12) carries an inline Verify block naming the specific unit/integration tests and end-to-end tests that prove it, each with a named red-before-green failure injection. §13 consolidates these into a single component→test→observability matrix, the structured-log contract, the red-before-green protocol, and a boundary/fuzz catalog. No behavior is described anywhere without the test that proves it.

Scope of what we verify: we trust Cue (Etheria's voice library) and do not re-test its internal recognition quality. Our tests verify only our integration WITH the library — that words/events are wired correctly, that actions round-trip, and that our own deterministic code behaves. Tunable parameters (timeouts, suggestion cadence, thresholds, latency budgets, word lists) are env-tunable knobs with documented defaults (ENG-A-01), not hard guarantees we assert against — so a Verify block checks the wiring and shape, not a fixed numeric promise about the library.

Why this was a judgment call

The approved draft already had a strong test section (§11). The call was where verification lives: a single end-of-doc section (cheaper to write, easy to skim past) versus threading it through every section plus a consolidated matrix (more effort, but it makes "assume nothing works until a test proves it" structurally enforceable — you cannot describe a behavior and forget to say how it is proven, because the Verify block sits right under it).

Alternatives considered

A. Keep one end-of-doc test section (the draft's shape). Lower effort. Rejected: verification reads as bolted-on; easy for a later author to add a behavior without a matching test; the operating bar explicitly says weight effort toward verification.
B. Inline Verify per section + consolidated §13 matrix (chosen). Every behavior is co-located with its proof; the matrix gives a single auditable surface; the red-before-green injection is named per test so "the agent said it works" can never substitute for a red run. Chosen.
C. Separate standalone test-plan document. Rejected: splits the design from its proof across two files; drift risk; a verifier would have to cross-reference constantly.

Consequences

  • The doc is longer and the matrix (§13.2) is the densest part — intended.
  • Red-before-green is named per test, so a build is never "green" without a demonstrated red run first.
  • The log contract (§13.3) is part of the design, not a downstream concern — a no-context agent can rebuild any loop from one correlationId.
