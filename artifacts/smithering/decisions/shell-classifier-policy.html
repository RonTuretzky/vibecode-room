CUT: no shell-command classifier, no shell gating — run dangerously

ENG-A-12REQ-11Scope cut

Superseded / CUT. The prior decision built a deterministic, parse-based, code-only shell classifier (safety/shell-classifier.ts) that sub-classified every shell call into read-safe (ungated) vs. mutating/unknown (gated), deny-by-default, backed by a probe that the parser split compound/redirect/substitution as assumed. The classifier, its allow/deny policy, the ShellVerdict type, and that probe are all removed.

The decision

There is no shell classifier and no shell gating. The system runs dangerously by default — shell commands run to completion without per-command classification or read-back. The safety/shell-classifier.ts requirement is dropped; there is no read-safe vs. mutating sub-classification, no compound/redirect/injection gating policy. This is an explicit scope-cut.

Why cut

The classifier only existed to keep a gated Safe mode autonomous for harmless shell while still holding destructive shell for a read-back. With the per-action safety gate cut (ENG-A-08) and execution modes collapsed to a single run-to-completion "dangerous" mode (ENG-A-11), there is nothing for the classifier to gate for — every shell command already runs. Building a deterministic shell parser/allowlist to feed a gate that no longer exists is pure dead weight.

Where safety comes from instead

Safety, if and when it is ever needed, comes from sandboxing the whole process — isolating it — not from permission classification of individual shell commands. That is deferred (revisited later), not built for V0.

  • No-shell-gate test: harmless and destructive shell alike run to completion without classification or read-back (RBG: re-introduce a shell gate that holds rm -rf → the run-to-completion e2e fails).
  • No classifier present: assert there is no safety/shell-classifier.ts and no ShellVerdict in the build.
