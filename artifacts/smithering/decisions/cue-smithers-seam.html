The Cue↔Smithers seam: one owned, async, bidirectional module

ENG-A-03   Panopticon V0 engineering  ·  the novel integration / top risk  ·  docs/planning/03-eng.md §8

The problem

Cue (ambient observation + routing) and Smithers (durable process management) are two independent stateful systems. No prior art integrates exactly these two (PRD §9, eng-oss §6.2) — it is the top integration risk. Done carelessly, the seam leaks: Cue grows knowledge of Smithers run internals, Smithers grows knowledge of cue policies, the spawn call blocks the Cue loop and blows the 3 s budget, and the novel risk is smeared across the codebase with no single place to test it.

Decision

The seam is exactly one module (src/seam/), and it is asynchronous and bidirectional. Neither Cue nor Smithers knows the other exists; the seam is the only code that speaks both schemas (the Temporal lesson — the adapter owns the translation, eng-oss §3.3). 
actions OUT (Cue → Smithers):
  Cue MappedActionTool {type,target,payload}
    → ActionDispatcher (Hono, async — returns immediately)
    → SmithersClient.spawn/steer/pause/signal      // isomorphic to Temporal signal/startChild

run-events BACK (Smithers → Cue):
  Smithers streamRunEvents (SSE)
    → RunEventNormalizer  → ≤15-word summary (before TTS)
    → cue.send_observation  → drives voice-out coherence

Why these properties

PropertyWhy it is load-bearing
One moduleThe novel risk gets one home to probe (P-SEAM), trace, and test — not N scattered call sites.
Async dispatchSpawn must complete without blocking the Cue loop (AC4.3 ≤3 s); required by the three-plane model (ENG-A-01).
BidirectionalVoice-out coherence needs durable-state run-events flowing back into Cue as observations — not just fire-and-forget actions.
Neither side knows the otherKeeps Cue swappable (D2 adapter posture) and Smithers swappable (Temporal is the documented fallback).

Isomorphism that de-risks it (eng-oss §3.3/§3.4)

Temporal's signal/startChild map directly onto Smithers' steer/signal/fork. {type:"steer",upid:"atlas",payload} → client.signal(upid,"steer",payload) — exactly a Temporal client call. Fork may require a fresh seeded run + parentId lineage rather than a native fork; P-SMITHERS determines which.

Alternatives

Inline the seam at each call site. Rejected: the top risk would have no single test home; schema drift would surface everywhere.
Synchronous spawn. Rejected: blocks the Cue loop and violates the 3 s budget + the Hot plane guarantee.
Fire-and-forget (actions only, no run-events back). Rejected: voice-out would have nothing coherent to say about a running process.
One owned async bidirectional module (chosen). Chosen.

Verification

  • Unit: action-schema-match (dispatched action == real MappedActionTool shape; RBG: rename a field → fails); async-dispatch (mocked slow Smithers doesn't block the decision plane); SSE-reconnect (drop+restore → no duplicate observations); UPID↔window correlation survives a simulated Cue restart.
  • E2e: full spine drives a real durable run, spawn→confirm ≤3 s; durability-recovery (kill backend mid-run → resume from checkpoint, REQ-15); fleet-isolation (steer A → B byte-identical, REQ-8/13).
  • Third-party: P-SMITHERS + P-SEAM (both P0).
Top risk: P-SEAM has no prior art to validate against — the probe must exercise the full round-trip against real instances of both systems before any seam code is trusted.
